### Post 04 — Por Que a Maioria das Automações Falha

```
Você já tentou automatizar algo na sua empresa — e quebrou em 3 meses?

Não foi culpa sua. E provavelmente não foi culpa da ferramenta.

O motivo mais comum de falha que vejo ao assumir automações de terceiros:

1. Automatizaram o processo errado
   Colocaram robô em cima de processo ruim. Automação de processo quebrado = processo quebrado mais rápido.

2. A solução foi construída sem dono
   Ninguém da empresa entende como funciona. Quando algo muda, ninguém sabe o que fazer.

3. Dependência excessiva de uma ferramenta específica
   A versão mudou. A API foi descontinuada. Tudo para.

4. Zero testes antes do go-live
   O robô foi ao ar e só descobriram o erro quando o cliente reclamou.

Em 8 anos e mais de 300 automações implementadas, nunca precisei abrir um pacote de suporte corretivo. Zero erros reportados.

Por quê?

Porque antes de escrever uma linha de código, entendo o processo. Mapeio as exceções. Testo cada cenário. E entrego documentação que qualquer pessoa da empresa consegue seguir.

Automação não é mágica. É método.

Se a sua última tentativa não funcionou, fale comigo. A conversa é gratuita.

### Post 14 — MCP and the Future of Data Operations

```
Most data teams still operate on tickets.

Engineer gets a Slack message → creates a Jira ticket → picks it up next sprint → delivers two weeks later.

For analytical queries, ad-hoc investigations, documentation updates, and routine pipeline checks — that cycle is too slow and too expensive.

MCP (Model Context Protocol) changes this.

With MCP servers connecting an LLM to your data stack, the same workflow becomes:

→ Data consumer asks a question in natural language
→ Agent queries Snowflake, runs the dbt model, checks the lineage graph
→ Answer is delivered with the source data attached — in seconds, not days

I've built this in production:
- MCP server connected to Snowflake for self-serve analytics
- Automated data team workflow where agents handle ticket creation → code generation → PR review → deploy
- LLM-powered pipeline monitoring that explains anomalies in plain English instead of throwing an alert number

This is not a demo. It's running. It reduces routine engineering toil by 60%+ on the tasks it touches.
